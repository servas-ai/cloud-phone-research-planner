## 1. Why this note exists

The Round-1 and Round-2 threat model (`docs/threat-model.md`, `docs/research-notes/ml-adversary-models.md`) frames the adversary as a **detector** running on a target app: rule-based (L_AdvA), tabular-ML on probe-vector (L_AdvB), or deep-ML on sensor traces (L_AdvC). H6 (`plans/10-h6-adversarial-coevolution-addendum.md`) further frames the **spoofer** as a finite Stackelberg player over a pre-committed mitigation-set M.

A Deep-Research bundle supplied by the human partner (Gemini-CLI, 2026-05-15) maps the 2026 adversary ecosystem as it actually exists in industry and OSS. It surfaces three classes of capability that the current threat model does not explicitly name:

1. **Agentic operator capability** — the *operator side* (who is driving the bot) has shifted from static scripts to LLM agents that visually interpret the screen via Vision-Language Models and inject motion via ADB. Frameworks: AppAgent, AppAgentX, DroidRun (mobilerun), Ghost-in-the-droid.
2. **Motoric humanisation** — touch events generated by these agents are no longer point-to-point: Bezier-curve paths with Perlin-noise jitter and velocity profiling are mathematically each-time-unique, defeating signature-based motion classifiers. Algorithmic prior art: `vincentbavitz/bezmouse` (Python+xdotool, BSD), Human-Behavior-Simulation-Kit, Sensey.
3. **Autonomous network rotation** — the *x402 protocol* (HTTP 402 "Payment Required" revived, USDC on Base/Solana L2) lets agents pay per-request for fresh 4G/5G mobile-carrier proxies, with no KYC and no static API keys.

Alongside these capability axes, the bundle also describes evolutions in the **system-spoofing stack** (LSPosed-based DeviceSpoofLab fingerprint masking, PairIP-aware AppManager bypass, Zygisk-Next + Shamiko + SuSFS for kernel-level hiding, Purifire as an eBPF-class evasion engine) and infrastructure (Bare-Metal ARM cloud phones: GeeLark, RedFinger, LDCloud, DuoPlus, BitCloudPhone).

This note translates each cluster into the project's existing vocabulary, identifies what is **net-new vs already covered**, and seeds the actionable proposed-changes list in the companion addendum.

### 2.2 Detector ↔ Operator decomposition

The current `docs/threat-model.md` collapses two distinct adversary axes into one (Adversary Capability A/B/C). The Gemini bundle separates them:

| Axis | What it characterises | Examples in the bundle |
|---|---|---|
| **Detection-Capability** (already in plan: A/B/C) | The defender's inference model: rule, tabular-ML, deep-ML | XGBoost on probe-vector, CNN on sensor trace |
| **Operator-Capability** (NEW) | What the attacker uses to drive the device | Manual ADB script, LLM-on-accessibility-tree, LLM-on-screen-pixels |
| **Network-Capability** (NEW) | What the attacker uses to source IPs | Static residential, manually-procured 4G, x402-autonomous 4G/5G |

These three axes are **orthogonal**: a powerful detector (L_AdvC) facing a weak operator (manual script) is a different game from the same detector facing an L_AdvC-equivalent operator that visually drives the device and rotates carriers per request.

**The H6 game is currently parameterised only on Detection-Capability.** This note proposes that the Discussion section of the eventual paper, and the Threats-to-Validity table in H6 (`plans/10` §8), name Operator-Capability and Network-Capability as **external-validity dimensions** — the H6 result is conditional on a fixed operator and a fixed network capability.

### 2.3 Operator-Capability axis (O_A / O_B / O_C)

| Level | Operator-Capability | Identifying signature on the device | Probe-inventory exposure |
|---|---|---|---|
| **O_A** | Hand-coded script using `AccessibilityService` | Enabled accessibility service for non-disabled user; coarse touch timing | Probe **#59** `env.accessibility_services` fires; Probe **#57** `ui.touch_pressure` shows roboticism |
| **O_B** | LLM-driven, accessibility-tree-based agent (e.g. early AppAgent) | Enabled accessibility service + LLM-tempo (~1 s latency per action) | #59 fires; #57 shows LLM-pace |
| **O_C** | LLM-driven, **screen-vision-only via ADB injection** (Ghost-in-the-droid, DroidRun stealth-adb, AppAgentX) | **No accessibility service** active; touch events injected through `/dev/uinput` from outside the device | **#59 silent — false negative**; #57 still informative if motoric humanisation is naive |

**Key observation:** the **O_A → O_C transition makes Probe #59 (`env.accessibility_services`) progressively less informative.** Against an O_C-class adversary, only motoric / behavioural probes remain useful at this layer. This is a previously unstated probe-inventory degradation that the threat model should acknowledge.

### 2.4 Motoric humanisation as a probe-inventory pressure

The Gemini bundle catalogues four algorithmic building blocks used by modern agentic frameworks to make touch events indistinguishable from human input:

1. **Quadratic Bezier path planning** — every swipe is computed from two random control points; no two swipes have the same trajectory hash. Reference: `vincentbavitz/bezmouse` (Python, BSD, originally for xdotool; the math transfers to ADB `input swipe` event streams).
2. **Velocity profiling** — acceleration → terminal velocity → deceleration, modelled on motor-control literature. Reference: Human-Behavior-Simulation-Kit.
3. **Perlin-noise tremor** — a small pseudo-random gradient field added to the Bezier path produces the muscle-tremor signature of a real hand.
4. **Calibration to gesture thresholds** — `Sensey` and similar libraries publish the framework-internal thresholds for "is this a tap vs a fling vs a long-press?", letting agents inject events exactly at the right kinematic envelope.

> **Framing (security-auditor RR-2):** The four techniques above are catalogued as the adversarial counterpart to the `behavior.touch_kinematic_signature` probe candidate (§3.2), for the purpose of characterising what that probe must distinguish from organic human input. This is **not** guidance for building motoric-humanisation tooling. Each item is named at the level needed to motivate a detector design, not at the level of a reference implementation.

The existing inventory has Probe **#57 `ui.touch_pressure`** at severity = `trace` with the description "MotionEvent pressure, isTrusted, bezier timing". The description acknowledges Bezier timing as relevant but the probe is not characterised in enough detail to test against motoric-humanised input streams.

**This note does NOT propose adding a probe.** It proposes that a future H6 round consider extending Probe #57 (or adding a sibling probe `behavior.touch_kinematic_signature`) as a P_ext candidate. The companion addendum surfaces this as an open question; the actual probe addition would require Round-3 OSF amendment + feasibility pilot per `osf-preregistration.md` §9 Amendment 2 (CV < 30 %, Cohen's d > 0.8).

### 2.6 Play Integrity arms race (TrickyStore, Integrity Box, keybox rotation)

The bundle describes the state of the keybox-injection market in 2026: Tricky Store with KernelSU WebUI is the current goldstandard; valid hardware keyboxes (extracted from real devices) are injected into the attestation chain; Google revokes leaked keyboxes server-side; Integrity Box automates rotation among private (uncompromised) keyboxes.

> **F22 clarification (security-auditor RR-1):** for the avoidance of doubt, neither "private" nor "uncompromised" status is a legal provenance criterion under F22. The only F22-compliant keybox sources remain (a) TEE extraction from project-member-owned devices, IRB-documented; or (b) written manufacturer permission. A privately-traded, technically-uncompromised keybox is still legally compromised if its chain of custody traces to a leaked, stolen, or non-consensually-extracted source.

**This material is descriptive only.** Our F22 Legal-Gate (`plans/05-validation-feedback.md` line 271) already requires every keybox to come from (a) own-device extraction or (b) written manufacturer permission. The bundle's "private keybox black market" content does NOT alter that constraint. The only addendum-worthy implication is a **literature row** noting that the Google revocation cadence has accelerated through 2025-2026, which:

- strengthens the case for the F22 + F23 institutional-only split (a published "working keybox list" would be stale within weeks anyway), and
- supports the H6 §9 commitment that mitigation-trajectory artefacts are institutional-only.

**Nothing here is a SpoofStack content change.** The F22 Legal-Gate stays as the gating event.

### 2.7 x402 — autonomous network capability

The x402 protocol uses the HTTP 402 "Payment Required" status code as the discovery vector for pay-per-request mobile proxy access; the agent receives the 402, pays a USDC micro-amount on Base/Solana L2, and re-issues the request with credentials returned in the 402 body. Effect: per-request 4G/5G IP rotation with no account, no KYC, no API key. Sources cited in the bundle: `proxies.sx`, `eco.com`.

This is the **Network-Capability N_C** in the §2.2 decomposition. Its threat-model implication is:

- Network-layer probes (#5 IP/ASN, #61 JA4, #62 TCP stack fingerprint, #63 HTTP/2 frame pattern) remain technically valid — the **packet-level signatures** are unchanged. JA4 / p0f / TCP-stack signatures are properties of the *device*, not of the *proxy*.
- What changes is the **IP-reputation defence becomes far weaker** because the attacker can burn IPs per-request without inventory cost. This affects the *server-side adversary* (out of scope per `ml-adversary-models.md` §4), not the *on-device adversary* this project measures.

**Implication for the project:** x402 should be cited in the Discussion section of the paper as an external-validity caveat — "our detection-resistance heatmap measures on-device-only signal, and IP-reputation defences that complement it have eroded materially in 2026". It does NOT change DetectorLab or SpoofStack.

## 3. Net-new threat-model elements

After §2's crosswalk, three items are net-new and not already covered by F1–F41 or by H6's pre-committed M / P_ext sets:

### 3.1 Kernel-Hook Adversary Class (Purifire / eBPF / kprobe injection)

Existing adversary hierarchy (`ml-adversary-models.md` §5) is about the **detector's inference model** (A/B/C). It does not characterise the **adversary's interception capability** on the device.

A Kernel-Hook adversary intercepts system-calls from inside the kernel (eBPF, kprobes, or KernelSU-side mount hiding). This means *probes that read from user-space `/proc/*`, `/sys/*`, `getprop`, or syscalls themselves* can be made to return forged values **without leaving a user-space artefact**. Defences against this require either:

- a probe that operates above the kernel (e.g. **hardware-attested measurement** — Probe #6 already covers this), or
- a probe that observes **timing-side-channel divergence** between forged and authentic execution paths (e.g. Probe **#64 `runtime.syscall_latency_histogram`**, added in Round-1 F5; the Roy Dipta 2024 EuroS&P paper already cited in `literature-extensions.md` row 22 is the published reference).

**This is not new probe work — both #6 and #64 already exist.** What is new is **explicitly naming Kernel-Hook adversary** as a STRIDE-adjacent row in `docs/threat-model.md` and updating the probe-mitigability table to reflect that probes #1, #7, #9, #28, #30, #14, #67 are *partially* mitigable by a Kernel-Hook adversary while #6, #64, #71 (binder cost) are *robust*. This is a refinement of the existing threat-model document, not a new experimental design.

**This material additionally informs H6:** the H6 mitigation-set M (`plans/10` §4.1) currently has M01–M08, none of which is a kernel-hook mitigation. The H6 Threats-to-Validity table should acknowledge that the H6 equilibrium is measured **within an adversary that does not have kernel-hook capability**. This is consistent with the current Stackelberg framing (M is finite and pre-committed) and does not require an OSF amendment, only a clarification in §8 of `plans/10`.

### 3.2 PairIP-Resident probe candidate

`com.pairip.licensecheck` is a Google-shipped native-hook detector that lives in many apps with Play-Console license enforcement. It uses native VM hooks to detect injected modules (LSPosed, Frida) at runtime. **Modern bypass works by surgically disabling the `ContentProvider` it relies on, not by hiding the hooks themselves.**

Our inventory has Probe **#60 `integrity.app_signature`** at severity = `trace` for "Tamper/hooking detection via package signatures", which is the *defender-side* version. We do not have a probe that asks the inverse question: **does the host system show evidence of PairIP-style detector tooling having been disabled?**

This is a candidate probe — **NOT proposed as an immediate addition** because:

- It must pass the §5.1-footer feasibility gate (CV < 30 %, Cohen's d > 0.8) per `plans/10` and `osf-preregistration.md` Amendment 2.
- It is meta-detection (detector-of-detector-of-detector), with subtle scope-implications.
- It would need to be in P_ext, not in the baseline-60, because it was not in the pre-registered v1.0 inventory.

The companion addendum lists this as one of two motoric/integrity candidate probes for a hypothetical Round-3+ OSF amendment, gated on the same feasibility pilot as Probe #75.
