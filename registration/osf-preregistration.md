## 3. Describe the key dependent variable(s) specifying how they will be measured.

| Variable | Operational definition | Measurement instrument |
|---|---|---|
| **DetectorLab probe-score** | Continuous score ∈ [0.0, 1.0] per probe per run, where 0.0 = "real device indistinguishable" and 1.0 = "container certain". | DetectorLab Kotlin app, Probe-Schema v1 JSON output (`docs/probe-schema.md`). One score field per probe, total 60+ probes per run. |
| **Per-config aggregate score** | Weighted mean of probe-scores per configuration, weighted by probe-criticality (Critical/High/Medium/Low) per `probes/inventory.yml`. | Computed offline in `experiments/aggregate/{config-id}.csv` (`plans/03-experiment-matrix.md` lines 47–54). |
| **Layer-transition delta** | Δ-score between L_n and L_{n+1} for each probe, paired by run-ID. | Paired-difference file generated by `experiments/runner/`. |
| **Coherence-score** (per A2) | Multivariate consistency metric over the probe-vector — concrete form (Mahalanobis / Mutual-Information / covariance-eigen-spectrum) locked in `docs/research-notes/coherence-metric-spec.md` per A2 (`plans/07-round-2-feedback.md` lines 109–113) **before** data collection. Pre-committed to whichever form the statistician selects (see §12). | Offline post-aggregation script. |
| **L_AdvB classifier output** | Predicted label `{real, spoofed}` and probability ∈ [0,1] from an XGBoost classifier trained on the probe-vector. | Offline scikit-learn / XGBoost pipeline; **not on-device** per cross-artifact resolution in `plans/07-round-2-feedback.md` lines 63–67. |

All scores are derived from probe-output JSON validated against `docs/probe-schema.md` v1; schema-validation failures are quarantined per §6.

## 5. Specify exactly which analyses you will conduct to examine the main question/hypothesis.

Per `plans/03-experiment-matrix.md` lines 24–32 and F3 (`plans/05-validation-feedback.md` lines 59–66):

1. **Per-probe layer-transition test (H1–H4):** McNemar paired test on binary "detected / not-detected" between adjacent configurations (L_n vs. L_{n+1}), paired by run-index. Threshold "detected" = score ≥ 0.40 (F13, `plans/05-validation-feedback.md` lines 187–192; thresholds 0.10/0.40 pre-registered).
2. **Multiple-testing correction:** Benjamini-Hochberg FDR at q = 0.05 across ~420 tests (60+ probes × 7 transitions). FDR > FWER rationale in §11.
3. **Effect-size:** Cohen's *g* for paired binary outcomes; 95% bootstrap CIs (10 000 resamples) on per-probe means.
4. **Layer-interaction:** ANOVA on the 2^6 Resolution-IV design — main effects + 2-way interactions for L1..L6.
5. **Coherence analysis (H5):** metric from `coherence-metric-spec.md` (A2). Configurations flagged "incoherent" if metric exceeds C0_real bootstrap 95th percentile.
6. **L_AdvB ML eval (H5):** offline XGBoost on probe-vectors, 10-fold CV, primary F1, secondary ROC-AUC and per-class precision/recall. Trained on labeled {C0_real, C1_L0a, …, C8_full}. Held-out 20% per label-class. Compared against L_AdvA static-rule baseline (threshold 0.40).
7. **Inter-probe correlation:** Pearson matrix per configuration. Probes with |r| > 0.85 in C0_real flagged redundant (`plans/03-experiment-matrix.md` line 29); PCA or domain-grouped scoring applied per F3 line 64.

All analyses scripted in Python (numpy / scipy / statsmodels / scikit-learn / xgboost), seed = 42, in `experiments/analysis/` (to be created).

## 6. Describe exactly how outliers will be defined and handled.

| Outlier class | Definition | Handling |
|---|---|---|
| **Schema-validation failure** | JSON output fails `docs/probe-schema.md` v1 schema validation. | Run quarantined to `experiments/runs/{config-id}/quarantine/`; **not** included in analysis; replaced by re-run from clean snapshot per `plans/03-experiment-matrix.md` lines 36–44. |
| **ABORTED_DRIFT run** | Container image hash, Magisk module versions, or keybox revocation status changed between snapshot creation and run completion (per F20, `plans/05-validation-feedback.md` lines 261–267, and `experiments/runner/SPEC.md` per Round-2). | Run aborted, snapshot rebuilt from manifest, re-run from clean state. Pre-abort partial data deleted. |
| **Statistical outlier (per-probe score)** | Score > 3 IQR above or below the per-config median for that probe. | Reported in supplementary materials but **retained** in primary analysis (we do not censor). Sensitivity analysis with outliers removed is reported separately. |
| **Coherence-metric outlier** | Per A2 spec — final definition pending statistician input. | Handled per A2 spec; default policy: report and retain unless schema/drift caused. |
| **Hardware fault** | ARM64 host crash, LTE modem disconnection > 30s, kernel panic. | Discard affected run; re-run from clean snapshot. Documented in `experiments/runs/{config-id}/incidents.log`. |

A run is "valid" iff: (a) schema-validates, (b) image-hash matches manifest, (c) all 60+ probes returned a score (no NaN), (d) DetectorLab process exited 0. Quarantined runs are reported but excluded from N counts.

## 7. How many observations will be collected or what will determine sample size?

**N = 60 runs per configuration** per F3 correction in `plans/05-validation-feedback.md` lines 59–66 (revised upward from the original N=30 in `plans/03-experiment-matrix.md` line 27).

**Total runs:** 8 monotone configurations × 60 + 16 fractional-factorial configurations × 60 = **1,440 valid runs**. With an expected 5% drift/quarantine rate, we plan for ≈ 1,520 attempted runs.

**Power-calculation justification:**
- McNemar paired test at α = 0.05 (two-sided), desired power 1-β = 0.80, target detectable effect size = 20-percentage-point difference in detection-rate between adjacent configurations (e.g. 80% → 60%) — this is the smallest effect the heatmap would treat as a meaningful "layer-mitigation success" per `plans/03-experiment-matrix.md` Aggregate row.
- Required discordant pairs: ≈ 50 per cell. To obtain ~50 discordant pairs at expected discordance-rate ≈ 0.85 (drawn from pilot estimate in `plans/03-experiment-matrix.md` lines 11–19), N ≈ 60 runs per configuration is required per `plans/05-validation-feedback.md` line 56.
- After Benjamini-Hochberg FDR-correction at q = 0.05 across 420 tests, the effective per-test α ≈ 0.0119 in the worst case; N=60 retains power ≥ 0.70 at the 20-pp effect size, which is acceptable for the exploratory layer-wise analysis.
- For H5 ML evaluation: 60 runs per configuration × 8 configurations = 480 labeled probe-vectors; with 10-fold CV this gives 48 test samples per fold per class — adequate for XGBoost on ≈ 60 features (rule-of-thumb ≥ 10× samples per feature for tabular ML).

**Stopping rule:** N = 60 per cell is hard-locked. We do **not** continue collecting until significance is reached, and we do **not** stop early upon significance. Total budget ≈ 1,520 runs × ~5 minutes per run = ~127 hours of orchestrator wall-clock time, planned for Phase 4 (Weeks 7–10) per `plans/00-master-plan.md` Phase 3–5 Gantt.

## 8. Anything else you would like to pre-register?

**Stop conditions** (any one triggers re-baselining, which means: rebuild snapshot, document cause, restart that configuration's N=60 from zero):
1. **Keybox revocation mid-study** (Google Play Integrity backend rotates keybox set, invalidating L3) — re-baseline L3, L4, L5, full-stack configurations.
2. **Container image hash drift** (a Docker pull, Magisk module auto-update, or kernel patch alters the manifest) — re-baseline affected configuration only.
3. **Magisk / Zygisk / LSPosed module version change** — re-baseline L0b and downstream.
4. **DetectorLab APK version change** — re-baseline all configurations including C0_real (Magisk auto-update is disabled per F20 in `plans/05-validation-feedback.md` line 266).
5. **OSF amendment** required for any pre-registered hypothesis change — see §9 and §12.

**Lab-isolation verification** (per F24 in `plans/05-validation-feedback.md` lines 314–322): before each batch run, the container-egress iptables policy is verified to drop all outbound traffic except via LTE-NAT. Any verification failure halts data collection.

**Run determinism artifacts** (per F20 and Round-2 `runner/SPEC.md`): every run produces a BLAKE3 run-ID over (image-hash, manifest-hash, APK-hash, run-timestamp) to permit forensic cross-check.

## 11. Researcher Degrees of Freedom Acknowledged

Per AsPredicted format §11, we explicitly enumerate the choice-points where a different defensible analyst might have chosen otherwise, with rationale for each:

1. **Choice of N = 60 per configuration** — rationalized via the power calculation in §7: N = 60 is the minimum to achieve power ≥ 0.80 for McNemar at the 20-pp effect-size after BH-FDR correction at q = 0.05 across 420 tests. A more conservative N = 100 was considered but exceeds the 12-week phase-budget per `plans/00-master-plan.md` Gantt (Phase 3–5).
2. **Choice of FDR (Benjamini-Hochberg) over FWER (Bonferroni)** — rationalized: FDR is the appropriate correction for exploratory layer-wise analysis where the cost of a false-positive (one extra "this layer matters" claim) is much smaller than the cost of a false-negative (missing a real layer effect). Bonferroni at α = 0.05 / 420 = 1.2e-4 would be massively underpowered at N = 60.
3. **Choice of coherence-metric form (Mahalanobis / Mutual-Information / covariance-eigen-spectrum)** — rationalized via A2 (`plans/07-round-2-feedback.md` lines 109–113): final form to be locked by the statistician in `docs/research-notes/coherence-metric-spec.md` **before** data collection. We pre-commit to whichever form is selected in A2 — this prevents post-hoc metric-shopping.
4. **Choice of fractional-factorial Resolution IV (16 cells, instead of full 2^6 = 64-cell factorial)** — rationalized: full factorial would require 64 × 60 = 3,840 runs, exceeding the orchestrator wall-clock budget by 2.7×. Resolution IV preserves identifiability of all main effects and main-effect-vs-2-way-interaction confounding only; 3-way interactions are aliased but academically uninteresting.
5. **Choice of detection threshold 0.40** for binary "detected" classification (from F13 in `plans/05-validation-feedback.md` lines 187–192) — rationalized: this threshold is the consensus midpoint between "near-zero noise floor" (0.10, used for "indistinguishable from real") and "high-confidence detection" (0.70, used for L_AdvA in the heatmap). Sensitivity analysis at thresholds 0.30 and 0.50 is reported as supplementary.
6. **Choice of XGBoost over LightGBM / RandomForest for L_AdvB** — rationalized: XGBoost is the most-cited tabular-ML baseline in the fraud-detection SDK literature (`docs/research-notes/ml-adversary-models.md` §1 lines 10–22; ATLAS 2025 baseline per §3 line 51). LightGBM and RandomForest are reported as supplementary robustness-checks.
7. **Choice of α = 0.05 (not 0.01 or 0.001)** — rationalized: standard for security-research empirical work at USENIX/NDSS; α = 0.01 was considered but combined with FDR-q = 0.05 yields the same effective FDR control.

## 12. Open Decision Points (for amendment if pilot data demands)

Decision-points that may require an OSF amendment (with explicit sign-off, timestamp, and change-log) if pilot data reveals an unanticipated obstacle:

1. **Coherence-metric mathematical form** — Pending A2 deliverable. If the statistician selects a metric not envisioned in §3 (e.g. some divergence other than Mahalanobis / MI / covariance-eigen-spectrum), this document is amended **before** data collection.
2. **Probe #75 inclusion** — Pending feasibility-pilot per Amendment 2 in §9. If the pilot fails the acceptance criterion, the probe is dropped and the change documented as an addendum here (not as a "switched analysis" — it never enters main data).
3. **Keybox availability** — If F22 legal-clearance restricts keybox provenance to "own-device-extraction only" (per `plans/05-validation-feedback.md` lines 167–171, Option B), and own-device extraction proves technically infeasible within Phase 0, then H2 is pre-flagged as **"unfalsifiable in this study"** per F11 Option C (`plans/05-validation-feedback.md` line 170). H2 results would then be reported as a negative finding ("L3 not testable") rather than dropped silently.
4. **L_AdvB training-corpus coverage** — If the labeled training corpus for the offline XGBoost classifier proves insufficient (< 480 valid probe-vectors after quarantine), H5 evaluation is reported with reduced power and explicitly caveated; the hypothesis is **not** restated.
5. **2^6 fractional-factorial cell collisions** — If two cells in the fractional-factorial map to the same physical configuration as a monotone cell (e.g. all-on = C8_full), data are pooled with explicit notation; we do **not** treat the same N=60 as N=120 for the same cell.
6. **DetectorLab schema v1 deprecation** — If a critical bug requires Probe-Schema v2 mid-study, all prior runs are re-validated against v2 if backward-compatible, otherwise they are quarantined and re-collected. Schema-version per run is recorded for audit.
7. **Real-device sensor-trace IRB scope** — Per `docs/research-notes/ml-adversary-models.md` Open Question 2 (line 114), if IRB declines coverage for the n=5-lab-phone sensor-trace baseline, H3 falsification (A/B trace-replay test) cannot be performed; H3 is reported as "untested due to IRB scope" rather than re-formulated.
